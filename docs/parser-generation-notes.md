# Parser Generation Notes

## Eberron: Forge of the Artificer review

Use these rules when tightening deterministic Aurora XML generation:

- When a source declares a 2024 Player's Handbook dependency, prefer `ID_WOTC_PHB24_*` spell and feat IDs over legacy `ID_PHB_*` IDs. Include renamed spell mappings such as Branding Smite to Shining Smite, and verify edge cases such as Melf's Acid Arrow.
- Background ability score increases should use `Ability Score Improvement` combination grants and also grant `ID_INTERNAL_GRANTS_BACKGROUND_ASI`, matching the 2024 core background pattern.
- Damage resistances should be emitted as `grant type="Condition"` entries, not `Condition Immunity`.
- When Aurora stat expressions do not support multiplication cleanly, emit repeated stat entries for level-scaled totals, such as five `level:artificer` entries for five times Artificer level.
- Companion stat blocks should be represented as real `Companion` elements with `traits`, `actions`, and `reactions` setters, plus child `Companion Trait`, `Companion Action`, and `Companion Reaction` elements.
- Spell `<supports>` values should stay focused on class/tags. Spell level belongs in setters/rules, not as an extra supports token.
- Avoid placeholder-shaped `ID_*` text in XML comments because validators that scan raw text can treat comments as unresolved ID references.
- Dragonmark feats use `<supports>Dragonmark</supports>`, while the Boon of Siberys uses `<supports>Epic Boon</supports>`. Prepared spell grants should come from explicit "always have ... prepared" wording. Full "Spells of the Mark" expansion tables should become separate `Feat Feature` children with `<spellcasting all="true" extend="true">` entries, then be granted from the parent dragonmark feat.
- 2024 background `short` setters should use canonical phrasing: `Feat` suffixes, joined skill pairs such as `History and Survival Skills`, and singular choice labels such as `one Artisan's Tool` even when the select `supports` value remains `Artisan tools`.
- Some canonical helper rules are source-text driven (`Experimental Elixir`, `Adventurer's Atlas`, `Superior Atlas`), while a few require known named-feature shapes because source text can omit the stat block details (`Eldritch Cannon`, `Steel Defender` companion select). Keep those helpers narrow and named-feature based.
- Greater Mark of Hospitality should choose its Intelligence/Wisdom/Charisma modifier through `Feat Feature` helper elements, with only the proficiency component as a parent stat rule. Potent Dragonmark uses `value="level:half:up" maximum="5"` for the extra slot level.
- The EFA corpus benchmark currently reaches 101/102 exact generated-to-canonical element shape matches with 0 unmatched generated elements and 0 high-severity semantic diffs. The remaining mismatch is intentionally not generalized yet: Homunculus Servant uses `conjuration, ritual` keywords plus explicit `isConcentration=false`, while the general 2024 spell convention remains sparse and uses upcasting keywords when higher-slot text is present.
- PDF text layout is shared by the browser and corpus CLI. It preserves row-first ordering by default and switches to left-column then right-column ordering only when positioned text has at least 40 nonblank items, a qualifying interior x-gap of at least 18 points, and at least 25% of the items on each side. Keep raw layout rows separate from prose for table parsing; do not reassemble cross-column tables from flattened descriptions.
- Fizban's spells table binds left-column level/name/school rows to y-aligned right-column concentration/ritual/class rows. This metadata is attached to parsed spells separately from description prose, so table text cannot contaminate descriptions or create false spell headings.
- Browser and CLI page-range handling share the same strict parser. Malformed ranges and ranges that select no pages fail instead of falling back to a full-source scan.
- Fizban 2021 and Tasha's 2020 canonical spell XML corroborate a sparse legacy setter style: omit keywords, false ritual/concentration setters, and empty material setters; retain explicit verbal/somatic/material component booleans. Unknown older sources retain the explicit legacy style until comparable canonical evidence supports another policy.
- Preserve source material wording in generated setters. Corpus comparison treats only `500 gp` versus `500gp` material-cost spacing as equivalent, because the Fizban canonical file compresses that unit spacing while the PDF does not.
- A bounded continuation policy supports selected PDF ranges that end in the middle of an entity: the selected pages remain the parser input, while the immediately following unselected page is supplied only as context. The spell parser uses that context when the selected-page spell body is empty, begins at direct spell prose such as "You...", and stops at the next section/element boundary. For selected pages that stop at page chrome, the parser may continue from the next selected page and can begin at named benefit prose such as `Breath Weapon.`. Reports disclose every unselected continuation page read.
- Fizban spell prose cleanup removes verified non-source artifacts from descriptions: page headers/footers, OCR sidebar blocks, and the spell-list table's class/concentration/ritual column text. It preserves actual spell effect tables such as Nathair's Mischief's `d4 Effect` table. Source validation now warns if page chrome, spell-table class-column text, or obvious sidebar OCR fragments leak into captured descriptions.

## Future parser work

- Treat [source-validation-roadmap.md](source-validation-roadmap.md) as the validation target for the next larger milestone: source-faithful normalized data first, Aurora-native XML shape second, and first-party comparison/reporting before calling a source complete.
- Deterministic feat and background rule generation now uses structured rule records and renders concise XML comments next to inferred rules, for example `<!-- Source rule: Increase your Strength score by 1. -->`. Extend the same pattern to class, subclass, race/species, companion, and magic item rules as those generators become more canonical.
- Source ruleset selection is deterministic and should remain auditable: publications with an explicit year of 2024 or later, or sources containing high-confidence 5.5e-only terminology such as Origin Feat, Epic Boon, Weapon Mastery, Mastery property, Magic action, Utilize action, D20 Test, Heroic Inspiration, 2024 upcast wording, Emanation ranges, or Bastion-specific rules, use 2024 wording and rules. Unknown or older sources default to 2014. Avoid broad signals that also exist in 2014, such as Search action by itself. Preserve the actual publication year when a 5.5e signal appears in older/unknown metadata; record the signal in ruleset evidence rather than rewriting the year.
- Generated 2024-style backgrounds should continue to mirror PHB24 canonical shape: compact metadata list in the description, optional feat reference div, ASI combination grant, feat/proficiency grants, `ID_INTERNAL_GRANTS_BACKGROUND_ASI`, then choice selects. Unknown/2014 sources should not use PHB24 background ASI combination grants.
- Generated spells should be edition-aware: 2024 spells stay sparse like canonical PHB24 spell XML, omit empty keyword/material setters, omit false boolean setters, and may infer PHB24-only support tags such as `Damaging Spell` or `Healing Spell`. Verified 2020-2021 canonical sources also use sparse legacy setters while retaining component booleans. Unknown/2014 sources should keep the older explicit setter shape and avoid PHB24-only support tags. Upcast wording is ruleset-sensitive: 2024 uses `Using a Higher-Level Spell Slot.`, while 2014 uses `At Higher Levels.`
- Generated race/species ability score stat rules should include `requirements="!ID_INTERNAL_GRANTS_BACKGROUND_ASI"` so they do not stack with 2024 background ASI choices.
- Keep expanding `scripts/benchmark-corpus.js` against extracted text/Markdown sources and canonical Aurora folders before adding broad heuristics. Exact-match gains should come from reusable XML shape rules; one-off canonical inconsistencies should remain notes until another source confirms the pattern. Treat missing IDs, grants, choices, and stats as high-severity semantic diffs even when the total exact-match percentage remains high.
- PDF-backed corpus entries should enforce minimum extraction/match thresholds so a source that extracts zero elements cannot pass accidentally. Use active PDF smoke entries for "no high-severity semantic diffs" and stricter Markdown/text entries for near-exact shape thresholds until the PDF parser is less noisy.
- When benchmarking official Aurora supplement folders, use canonical-style abbreviations such as `WOTC_FTOD`, `WOTC_ERLW`, or `WOTC_GGTR`; otherwise ID-prefix differences hide the real semantic comparison. Do not auto-prefix normal user-entered homebrew abbreviations with `WOTC_`.
- Real PDF passes exposed recurring false positives: prose headings followed by "You have..." can be mistaken for feats, and OCR/table text can be mistaken for magic items, archetypes, or background features. Prefer local block-shape guards over broad title blacklist rules.
- ERLW and Guildmasters noise split into three buckets: parser false positives from loose title/prose detection, OCR line contamination such as `FEATURE: HOUSE CONNECTIONS about...` or `FEATURE: LEGION STATION Cantrip...`, and real generation gaps. Reusable parser fixes should preserve the EFA strict corpus: section-aware feat boundaries, magic item category validation, title-fragment guards, canonical background feature IDs, `Feature:` background feature display names, and `Background Feature` support tags are safe; 2014 background personality-list selects, guild spell-list features, and named feats such as Revenant Blade still need dedicated generation work.
- The EFA source gate now validates class and 2024 species/race source coverage. The near-exact XML benchmark includes species/race after matching the first-party EFA shape: `ID_*_RACIAL_TRAIT_<RACE>_<TRAIT>` IDs, non-repeated IDs for traits already prefixed by the species name, Small/Medium size selects, internal race grants, Common language defaults, and text-driven trait rules for skill choices, resistances, darkvision, AC bonuses, Khoravar spellcasting ability helpers, and Shifter form helpers. It also includes the Artificer class/class-feature XML shape: core trait table parsing, canonical `ID_EFA_CLASS_FEATURE_ARTIFICER_*` feature IDs, table-derived feature grants, multiclass rules, spellcasting progression stats, Replicate Magic Item plan/item stats, ASI and Epic Boon selects, attunement limits, and filtered placeholder subclass rows.
